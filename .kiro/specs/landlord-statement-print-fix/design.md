# Landlord Statement Print Fix — Bugfix Design

## Overview

The "Print / Save PDF" button on the Landlord Statement detail page (`/finance/statements/:id`) silently
fails to open the browser print dialog. The popup window opens and receives the HTML content, but
`print()` is never called.

The root cause is a classic race condition in `handlePrint` inside
`src/pages/finance/StatementDetailPage.tsx`: `printWindow.onload` is assigned **after**
`printWindow.document.close()` has already been called. When the document contains no external
resources (no logo image), it loads synchronously, so the `load` event fires before the listener
is registered and the callback is silently dropped.

The fix eliminates the external `onload` listener entirely. Instead, a small inline `<script>` tag
is injected into the HTML by `generateStatementHTML()` — controlled by a new `autoPrint` parameter.
Because the script runs inside the popup's own execution context after the DOM is ready, it is
immune to the race condition and works consistently across all browsers.

---

## Glossary

- **Bug_Condition (C)**: The condition that triggers the bug — `handlePrint` assigns `printWindow.onload`
  after `document.close()` has already fired the `load` event, so `print()` is never called.
- **Property (P)**: The desired observable outcome for any `handlePrint` invocation — the browser print
  dialog must appear in the popup window.
- **Preservation**: All existing behaviour that must remain byte-for-byte identical after the fix:
  `handleView` must not trigger `print()`, popup-blocked toasts must still appear, and all HTML
  content sections must remain unchanged.
- **`generateStatementHTML`**: The function in `src/utils/statement-html.ts` that builds the complete
  self-contained HTML string written to the popup window.
- **`StatementPrintData`**: The interface (also in `src/utils/statement-html.ts`) that defines the
  parameters accepted by `generateStatementHTML`.
- **`handlePrint`**: The async function in `StatementDetailPage.tsx` that fetches company settings,
  calls `generateStatementHTML`, opens a popup window, and writes the HTML to it.
- **`handleView`**: The sibling async function that opens a popup for on-screen viewing without
  printing.
- **`autoPrint`**: The new optional boolean field added to `StatementPrintData` (default `false`).
  When `true`, `generateStatementHTML` appends an inline print script to the HTML `<body>`.
- **`printWindow.onload`**: The external event listener assignment in the current buggy code that is
  registered too late and is therefore never called.

## Bug Details

### Bug Condition

The bug manifests when `handlePrint` opens a popup window, writes inline HTML to it (with no
external resource URLs), and closes the document. Because no network fetches are required, the
browser loads the document synchronously during `document.close()`. The `onload` listener assigned
immediately afterwards is therefore registered against an already-loaded document, so its callback
never fires and `window.print()` is never called.

**Formal Specification:**

```
FUNCTION isBugCondition(X)
  INPUT: X of type PrintAttempt {
    popupAllowed: boolean,
    documentLoadedBeforeListenerSet: boolean
  }
  OUTPUT: boolean

  RETURN X.popupAllowed = true
         AND X.documentLoadedBeforeListenerSet = true
END FUNCTION
```

`documentLoadedBeforeListenerSet` is `true` when either:
- The HTML contains no external resource URLs (no logo), so the document loads synchronously; OR
- The browser completes an image fetch before the JS engine returns from `document.close()` (less
  common but theoretically possible for cached images).

### Examples

- **No logo, popup allowed** — popup opens, HTML written, `document.close()` fires synchronous load,
  `onload` registered, never called, print dialog never appears. *(Bug triggered)*
- **Logo URL present, popup allowed** — popup opens, HTML written, image fetch begins asynchronously,
  `document.close()` called, `onload` registered, image load completes, `onload` fires, print dialog
  appears. *(Bug not triggered — asynchronous load wins the race)*
- **Popup blocked** — `window.open` returns `null`, early-return guard triggers, toast shown. *(Bug
  not applicable)*
- **After fix — no logo, popup allowed** — inline `<script>` runs inside popup after DOM is ready,
  `window.print()` called after 300 ms `setTimeout`. *(Fixed)*

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- `handleView` must open a popup displaying the statement HTML without triggering the print dialog.
- Popup-blocked paths must continue to call `showError(...)` with the existing toast messages
  (`'Print blocked'` / `'View blocked'`).
- All HTML content sections generated by `generateStatementHTML` (landlord details, summary cards,
  property breakdown, rent transactions, expenses, footer) must remain identical.
- When a company logo signed URL is available it must continue to be embedded in the HTML.
- When company settings cannot be fetched, the function must continue to fall back gracefully and
  render the statement without a logo.

**Scope:**
All code paths that do NOT satisfy `isBugCondition` (i.e. `handleView`, blocked-popup paths, and
error paths) must be completely unaffected by this change.

## Hypothesized Root Cause

Based on code inspection of `handlePrint` in `StatementDetailPage.tsx`:

1. **Late `onload` Registration (Confirmed Root Cause)**: `printWindow.onload` is assigned on the
   line *after* `printWindow.document.close()`. The MDN specification states that `document.close()`
   signals the end of writing and may fire `load` synchronously when there are no pending resource
   loads. Once `load` has fired, assigning `onload` has no effect.

2. **No External Resources in Base Case**: `generateStatementHTML` produces fully inline HTML with
   no `<link>` or `<script src="...">` tags. When `companyLogo` is `null` (company has no logo),
   there is no `<img>` tag either, so the document is guaranteed to load synchronously.

3. **Logo Case Masks the Bug**: When a logo URL is present, the `<img>` fetch is asynchronous,
   creating a window during which `onload` can be registered before the event fires. This explains
   why the bug may appear intermittent.

4. **No Fallback Mechanism**: The current code has no fallback if `onload` is not called (e.g. a
   direct `setTimeout(() => printWindow.print(), ...)` after `document.close()`), so any missed
   `onload` event results in complete, silent failure.

---

## Correctness Properties

Property 1: Bug Condition — Print Dialog Must Appear

_For any_ `PrintAttempt` where `isBugCondition` returns `true` (popup is allowed and the document
loads before the external `onload` listener can be registered), the fixed `handlePrint` function
SHALL trigger the browser print dialog inside the popup window.

**Validates: Requirements 2.1, 2.2, 2.3**

Property 2: Preservation — View Flow Must Remain Unchanged

_For any_ input that does NOT satisfy `isBugCondition` — specifically any invocation of `handleView`,
any blocked-popup path, or any call to `generateStatementHTML` with `autoPrint: false` or no
`autoPrint` argument — the fixed code SHALL produce exactly the same observable behaviour as the
original code: no print dialog is triggered, existing toast messages are unchanged, and the generated
HTML is structurally identical.

**Validates: Requirements 3.1, 3.2, 3.3, 3.4, 3.5**

## Fix Implementation

### Changes Required

#### File: `src/utils/statement-html.ts`

**Interface update — `StatementPrintData`:**
1. **Add `autoPrint?: boolean`** — optional field, defaults to `false` when omitted.

**Function update — `generateStatementHTML`:**
2. **Destructure `autoPrint`** from `data`: `const { ..., autoPrint = false } = data`
3. **Inject inline print script** — immediately before the closing `</body>` tag, when
   `autoPrint === true`, append:
   ```html
   <script>
     window.addEventListener('load', function() {
       setTimeout(function() { window.print(); }, 300);
     });
   </script>
   ```
   The `load` listener fires after any pending image loads; the 300 ms `setTimeout` gives the
   browser a frame to complete layout before opening the print dialog. When `autoPrint === false`
   (default) nothing is appended, preserving the view-only path.

#### File: `src/pages/finance/StatementDetailPage.tsx`

**`handlePrint` changes:**
4. **Pass `autoPrint: true`** in the `generateStatementHTML(...)` call inside `handlePrint`.
5. **Remove the `printWindow.onload` block** — delete the entire block:
   ```ts
   printWindow.onload = () => {
     setTimeout(() => {
       try {
         printWindow.print()
       } catch (printErr) {
         console.error('Print dialog failed:', printErr)
         showError('Print failed', 'Could not open print dialog')
       }
     }, 500)
   }
   ```
   `print()` is now handled by the inline script; the external listener is no longer needed.

**`handleView` changes:**
6. **No change required** — `handleView` calls `generateStatementHTML` without `autoPrint`, so
   it will default to `false` and the print script will not be injected.

## Testing Strategy

### Validation Approach

Testing follows a two-phase approach:

1. **Exploratory phase** — run tests against the *unfixed* code to confirm the race condition is
   reproducible and to verify that the `handleView` path already works correctly (it does not
   depend on `onload`).
2. **Fix + Preservation phase** — apply the fix, then run fix-checking tests (Property 1) and
   preservation-checking tests (Property 2) to confirm correct behaviour and zero regressions.

### Exploratory Bug Condition Checking

**Goal**: Reproduce the race condition on the unfixed code to confirm the root cause before writing
the fix.

**Test Plan**: Mock `window.open` to return a fake popup object with a synchronously-loading
document. Call `handlePrint` and assert that `print()` was never called. This will fail on unfixed
code (confirming the bug) and pass after the fix.

**Test Cases:**

1. **Synchronous Load Test** — mock `document.close()` to immediately fire `load`; assert
   `print()` was *not* called on unfixed code. *(expected to fail on unfixed code)*
2. **No-Logo Synchronous Load** — call with `companyLogo: null`; the HTML has no `<img>` so
   load is synchronous; assert `print()` not called. *(expected to fail on unfixed code)*
3. **Logo Present, Async Load** — mock an async image load; assert `print()` *is* called after
   `onload` fires. *(may pass on unfixed code — documents the race-condition boundary)*
4. **Popup Blocked** — mock `window.open` returning `null`; assert `print()` never called and
   `showError` was invoked with `'Print blocked'`. *(expected to pass on both unfixed and fixed)*

**Expected Counterexamples:**
- `print()` is never called when the document loads synchronously (no logo).
- Possible causes: late `onload` registration, synchronous `document.close()`, no fallback.

### Fix Checking

**Goal**: Verify that for all inputs where the bug condition holds, the fixed `handlePrint` calls
`print()`.

**Pseudocode:**

```
FOR ALL X WHERE isBugCondition(X) DO
  result := handlePrint_fixed(X)
  ASSERT printDialogTriggered(result) = true
END FOR
```

**Test Cases (on fixed code):**

1. **Synchronous Load — No Logo** — mock synchronous `document.close()`; assert `print()` *is*
   called (via inline script timeout).
2. **Synchronous Load — With Logo** — HTML contains `<img>` but mock makes load synchronous;
   assert `print()` is still called.
3. **Delayed Load — With Logo** — mock async image fetch completing 200 ms after `document.close()`;
   assert `print()` is called after inline script fires.

### Preservation Checking

**Goal**: Verify that for all inputs where the bug condition does NOT hold, the fixed code produces
the same behaviour as the original.

**Pseudocode:**

```
FOR ALL X WHERE NOT isBugCondition(X) DO
  ASSERT handleView_original(X) = handleView_fixed(X)   // no print(), window opens with HTML
  ASSERT generateStatementHTML({ ..., autoPrint: false }) = generateStatementHTML_original({ ... })
END FOR
```

**Testing Approach**: Property-based testing is recommended here because:
- It can generate many combinations of `StatementPrintData` inputs automatically.
- It provides strong guarantees that `generateStatementHTML` output is stable for all non-`autoPrint`
  inputs.
- It catches unexpected HTML mutations that manual tests might miss.

**Test Cases:**

1. **View Flow Preservation** — call `handleView`; assert `print()` was never called and the popup
   receives the expected HTML.
2. **Popup-Blocked Preservation** — mock `window.open` returning `null` in both print and view
   paths; assert existing toast messages are unchanged.
3. **HTML Content Preservation** — call `generateStatementHTML` with `autoPrint: false` (or
   omitted) and compare output with the pre-fix baseline; assert no structural differences.
4. **HTML Content with autoPrint: true** — call `generateStatementHTML` with `autoPrint: true`;
   assert the output contains the inline print script immediately before `</body>` and is otherwise
   identical to the `autoPrint: false` output.

### Unit Tests

- Test that `generateStatementHTML` with `autoPrint: true` injects the print script before `</body>`.
- Test that `generateStatementHTML` with `autoPrint: false` (or omitted) does NOT inject the script.
- Test that `handlePrint` no longer assigns `printWindow.onload`.
- Test `handlePrint` popup-blocked path: `showError('Print blocked', ...)` is called.
- Test `handleView` popup-blocked path: `showError('View blocked', ...)` is called.
- Test edge cases: `companyLogo: null`, very long landlord names, missing optional fields.

### Property-Based Tests

- Generate random `StatementPrintData` objects and assert that the output of
  `generateStatementHTML({ ...data, autoPrint: false })` never contains the print script string.
- Generate random `StatementPrintData` objects and assert that the output of
  `generateStatementHTML({ ...data, autoPrint: true })` always contains exactly one print script
  block immediately before `</body>`.
- Generate random `StatementPrintData` objects and assert that the HTML output with
  `autoPrint: true` and `autoPrint: false` differ by exactly the injected script and nothing else
  (structural equality after stripping the script block).

### Integration Tests

- Simulate a full print flow from clicking "Print / Save PDF" through to `print()` being called,
  using a jsdom popup mock with synchronous document loading (no logo).
- Simulate a full print flow with a logo URL present (async image); assert `print()` is still called.
- Simulate a full view flow and assert `print()` is never called.
- Simulate popup-blocked scenarios for both print and view flows; assert correct toast messages.
